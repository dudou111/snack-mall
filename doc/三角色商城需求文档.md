# 三角色商城需求文档

## 1. 文档信息
- 文档名称：三角色商城需求文档
- 项目名称：零食商城（snack-mall）
- 版本：v1.0
- 日期：2026-04-22
- 适用范围：管理员端（react-oa）、用户/商家端（supermarks）、统一后端（server）与 MongoDB 数据同步

## 2. 背景与目标
当前项目已具备基础商品与订单能力，但前端角色视角和业务闭环尚不完整。目标是在现有架构上建立三角色协同闭环：
- 管理员：在 react-oa 管理系统中查看用户购买记录、商家上传记录，追踪订单物流与支付退款状态。
- 用户：在 supermarks 中浏览商品、下单、模拟支付、发起退款并查看订单状态。
- 商家：在 supermarks 中上传/管理商品，查看自身上传记录与相关订单履约信息。
- 全流程数据实时写入并同步 MongoDB，确保三端查看的数据一致。

## 3. 系统范围

### 3.1 本期范围（In Scope）
- supermarks 单项目多角色（用户/商家）能力建设。
- 管理员在 react-oa 的记录查看与订单追踪增强。
- 模拟支付（支付成功/失败）与退款（成功/失败）业务闭环。
- 订单物流状态流转（待发货 -> 配送中 -> 已送达；异常分支：配送失败）。
- 购买记录、上传记录、支付记录、退款记录统一入库并可追溯。

### 3.2 非本期范围（Out of Scope）
- 第三方真实支付网关接入（微信/支付宝签名、回调验签）。
- 第三方真实物流轨迹地图。
- 多租户财务结算与分账。
- 完整 RBAC 控制台（仅定义角色模型与接口权限边界）。

## 4. 角色定义与职责

### 4.1 管理员（Admin，react-oa）
- 查看全量购买记录（按时间、状态、用户、商品筛选）。
- 查看商家上传记录（按商家、时间、商品状态筛选）。
- 查看并更新订单状态、支付状态、物流状态。
- 查看退款申请并执行退款处理。
- 查看关键统计（订单数、支付率、退款率、商家上传量）。

### 4.2 用户（User，supermarks）
- 浏览商品、搜索/筛选、下单。
- 发起模拟支付（成功/失败）。
- 查看订单状态与物流状态。
- 发起退款申请并查看退款结果。

### 4.3 商家（Merchant，supermarks）
- 上传商品、编辑商品、上下架商品。
- 查看自身商品上传记录和商品变更记录。
- 查看涉及自身商品的订单并更新履约物流状态。

## 5. 现状映射（基于现有代码）

### 5.1 已具备能力
- 后端已有订单模型与状态字段：status/paymentStatus/deliveryStatus。
- 后端已有订单创建、列表、详情、状态更新接口。
- 后端已有商品 CRUD 与状态更新接口。
- 管理端已有订单列表和状态下拉更新页面。

### 5.2 待增强能力
- 用户模型仅有 isAdmin，缺少 role（admin/user/merchant）显式字段。
- 商品缺少 merchantId（商品归属商家）和上传审计字段。
- 订单缺少支付模拟结果流水号、退款申请信息、退款处理信息。
- 管理端缺少“商家上传记录”独立页面。
- supermarks 目录当前为空，需新建用户/商家端前端能力。

## 6. 业务流程

### 6.1 用户购买流程
1. 用户登录 -> 浏览商品 -> 加入购物车/立即购买。
2. 提交订单后创建订单记录（默认 status=待支付，paymentStatus=未支付）。
3. 用户发起模拟支付：
   - 成功：paymentStatus=已支付，status=待发货，写入 paymentTime。
   - 失败：保持未支付，记录失败原因。
4. 商家/管理员发货后：deliveryStatus=配送中，写入 shipmentTime。
5. 订单签收：deliveryStatus=已送达，status=已完成，写入 completionTime。

### 6.2 商家上传与履约流程
1. 商家登录 -> 上传商品（图片、价格、库存、分类、品牌）。
2. 系统记录上传人 merchantId 与操作时间。
3. 商家可编辑/下架商品，形成上传记录/修改记录。
4. 商家查看涉及自身商品订单，更新物流状态。

### 6.3 退款流程（模拟）
1. 用户在已支付且满足条件订单上发起退款申请。
2. 系统创建退款申请记录（申请时间、原因、金额、状态=待处理）。
3. 管理员审核：
   - 通过：paymentStatus=已退款，status=已取消（或退款完成），记录 refundTime。
   - 驳回：保留已支付，记录驳回原因。
4. 若订单未发货且退款成功，可回补库存。

## 7. 功能需求清单

### 7.1 管理员端（react-oa）
- FR-A01 购买记录管理
  - 支持分页、关键词搜索、时间筛选、状态筛选。
  - 可查看订单详情（用户、商品、金额、支付、物流、时间轴）。
- FR-A02 商家上传记录
  - 展示商家上传商品列表、创建时间、修改时间、状态、库存。
  - 支持按商家筛选与导出（CSV 可选）。
- FR-A03 订单追踪
  - 可更新订单 status、paymentStatus、deliveryStatus。
  - 状态更新需校验合法流转（见状态机约束）。
- FR-A04 退款处理
  - 查看退款申请列表、审批通过/拒绝、记录处理人和处理时间。
- FR-A05 统计看板
  - 订单总量、支付转化率、退款率、商家上传量、在售商品数。

### 7.2 用户端（supermarks）
- FR-U01 用户注册/登录与个人中心。
- FR-U02 商品浏览/搜索/筛选。
- FR-U03 下单创建订单并展示订单详情。
- FR-U04 模拟支付（成功/失败）并提示结果。
- FR-U05 订单查询与物流状态查看。
- FR-U06 退款申请（原因、金额、说明）与结果查看。

### 7.3 商家端（supermarks）
- FR-M01 商家登录与身份识别。
- FR-M02 商品上传（含图片上传）与校验。
- FR-M03 商品编辑、上下架、库存调整。
- FR-M04 上传记录查询（仅查看自身）。
- FR-M05 履约订单查询与物流状态更新。

## 8. 数据模型与字段要求

### 8.1 用户模型（User）
- 保留：username、password、isAdmin、status。
- 新增建议：
  - role: enum(admin,user,merchant)
  - merchantProfile（可选）：店铺名、联系人、联系方式

### 8.2 商品模型（Product）
- 保留：name、price、stock、status、category、brand、image/images。
- 新增建议：
  - merchantId: ObjectId（关联 User）
  - createdByRole: string
  - uploadTime / lastModifiedBy / lastModifiedTime

### 8.3 订单模型（Order）
- 保留：status、paymentStatus、deliveryStatus、paymentTime、shipmentTime、completionTime。
- 新增建议：
  - buyerId: ObjectId
  - merchantIds: ObjectId[]（订单商品所属商家）
  - paymentMockNo: string（模拟支付流水号）
  - refund: {
      applyTime, applyReason, applyAmount, reviewStatus, reviewBy, reviewTime, rejectReason, refundTime
    }

### 8.4 审计与日志（建议）
- 每次关键操作写入操作日志：operatorId、operatorRole、action、targetId、before/after、timestamp。

## 9. 状态机与流转规则

### 9.1 订单状态 status
- 待支付 -> 待发货 -> 配送中 -> 已完成
- 待支付 -> 已取消
- 待发货 -> 已取消（退款成功场景）

### 9.2 支付状态 paymentStatus
- 未支付 -> 已支付 -> 已退款
- 已退款不可逆转为已支付

### 9.3 物流状态 deliveryStatus
- 待发货 -> 配送中 -> 已送达
- 配送中 -> 配送失败（异常）

### 9.4 关键约束
- 未支付订单不可进入配送中。
- 已送达订单默认不可退款（可配置例外）。
- 退款通过后禁止再次退款。
- 同一订单状态更新需保证幂等（重复请求不产生重复副作用）。

## 10. 接口需求（在现有接口基础上扩展）

### 10.1 已有接口（可复用）
- 订单：
  - GET /api/order/list
  - GET /api/order/detail/:id
  - POST /api/order/create
  - PATCH /api/order/status/:id
- 商品：
  - GET /api/product/products
  - POST /api/product/products
  - PUT /api/product/products/:id
  - PATCH /api/product/products/:id/status

### 10.2 新增/增强接口（建议）
- 角色与身份
  - PATCH /api/user/role/:id（管理员设置角色）
- 商家上传记录
  - GET /api/product/upload-records?merchantId=&page=&pageSize=
- 支付模拟
  - POST /api/order/pay/mock
    - 入参：orderId, paymentMethod, mockResult(success|fail)
    - 出参：paymentStatus, paymentMockNo
- 退款申请
  - POST /api/order/refund/apply
  - GET /api/order/refund/list
  - PATCH /api/order/refund/review/:id（approve/reject）
- 物流跟踪
  - PATCH /api/order/delivery/:id
    - 入参：deliveryStatus

### 10.3 统一返回结构
- 成功：{ code: 0, message: string, data: any }
- 失败：{ code: 1, message: string, error?: string }

## 11. 权限与访问控制

### 11.1 接口权限矩阵
- admin：可访问所有管理接口。
- merchant：可访问自身商品与自身相关订单履约接口。
- user：可访问自身订单、支付、退款接口。

### 11.2 安全要求
- 所有业务接口必须鉴权（JWT）。
- 敏感操作（状态更新、退款审核）需记录操作日志。
- 严格校验越权：merchant 不能修改其他商家的商品与订单履约信息。

## 12. 数据同步与一致性要求
- 所有订单、支付、退款、物流、上传记录均落库 MongoDB。
- 同一订单在用户端、商家端、管理员端展示结果一致。
- 库存策略：
  - 当前沿用“下单即扣库存”。
  - 若退款成功且订单未发货，可执行库存回补。
- 写操作失败必须回滚或补偿，避免出现“状态已变更但日志未记录”情况。

## 13. 非功能需求
- 可用性：核心 API 成功率 >= 99.9%。
- 性能：常规列表接口 P95 < 500ms（万级数据需索引优化）。
- 扩展性：支付与物流模块预留真实网关扩展点。
- 可观测性：关键链路有日志与错误告警。

## 14. 验收标准（UAT）

### 14.1 用户侧验收
- 能正常下单并完成模拟支付。
- 支付后订单状态自动进入待发货。
- 能查看物流状态并发起退款申请。

### 14.2 商家侧验收
- 能上传商品并在记录中看到上传行为。
- 能查看涉及自身商品的订单并更新履约状态。
- 不能查看/操作其他商家数据。

### 14.3 管理员侧验收
- 能查看购买记录与商家上传记录。
- 能执行订单追踪与退款审批。
- 状态变更可在数据库中准确反映并多端一致。

### 14.4 异常场景验收
- 库存不足下单失败并提示。
- 未支付订单禁止发货。
- 重复退款申请被拦截。
- 越权请求返回权限错误。

## 15. 里程碑建议
- M1：角色模型与权限改造（后端）。
- M2：supermarks 用户/商家基础页面与下单流程。
- M3：模拟支付与退款流程。
- M4：管理员上传记录与追踪增强。
- M5：联调、压测、验收上线。

## 16. 风险与待确认事项
- 风险1：当前数据模型混合历史业务字段，改造时需避免影响既有功能。
- 风险2：下单即扣库存在高并发下需补充并发控制与幂等策略。
- 待确认1：退款是否允许部分退款（当前按全额退款设计）。
- 待确认2：已送达后是否支持售后退款窗口。

## 17. 结论
本需求文档定义了三角色商城从商品上传、购买下单、模拟支付、物流追踪到退款处理的完整闭环，并明确了数据库同步、权限边界与验收标准。按本文件实施后，可确保管理员在 react-oa 完成全局监管，用户与商家在 supermarks 完成业务操作，且全链路数据一致可追溯。
